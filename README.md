# Kenyan Merchant BTCPay PoS plugin

Open-source mobile Point of Sale for African merchants. Accept payments in both local mobile money (M-Pesa via a Payment Service Provider) and Bitcoin (Lightning). Two independent rails, two independent ledgers, no forced conversion.

Built on BTCPay Server, an embedded Lightning node on the merchant's phone, and Lightning Service Provider (LSP) protocol integration. Corridor-agnostic by design so any operator serving any African market can fork, configure their own LSP / PSP / rate source, and ship their own branded build.

**Status: early design. This README is the current spec. Code and detailed docs to follow.**

## Design principle

**Abstract Lightning away so merchants can just make sales.**

Merchants never need to think about channels, liquidity, routing, node health, or seed management during normal operation. All Lightning complexity is absorbed by the operating deployment (Tando PoS is the reference, or a forker's deployment) as an invisible operational responsibility. Anything that leaks Lightning terminology into the merchant experience is a design bug unless explicitly required for safety (e.g., seed backup).

## Overview

An open-source mobile PoS that accepts payments in two independent currencies:
- **Local mobile money** via a Payment Service Provider (e.g., Safaricom Daraja for Kenya)
- **Bitcoin** via Lightning, self-custodial on the merchant's phone

The merchant determines the final sale amount. The payer then chooses which rail to pay through at the moment of settlement, seeing both option prices side by side. The two rails settle to two independent ledgers, and there is no automatic conversion between them in v1.

The Tando team publishes the reference deployment ("Tando PoS") for the Kenyan corridor, using Bitcoin.co.ke as the LSP, Splice as the PSP (Daraja adapter), and a Trex-backed BTCPay rate plugin. The entire stack is open source (MIT) and designed to be forked and rebranded by any operator.

## Two-rail settlement model

The PoS accepts payments in two currencies that do not interact:

### KES rail (M-Pesa, aggregator model)
- Merchant registers their M-Pesa identifier (personal phone, Till Number, or PayBill Number) in settings
- Customer at the till supplies their M-Pesa phone number via a numeric keypad on the till device (customer or merchant taps)
- The operator's PSP (Splice, paybill 4141031, in the reference deployment) initiates an STK Push via Daraja against its own shortcode
- Customer approves on their M-Pesa app; funds settle into the PSP's shortcode
- The PSP triggers a B2C disbursement from its shortcode to the merchant's registered M-Pesa identifier
- Both events (STK Push confirmation and B2C settlement) are surfaced to the PoS via Daraja callbacks
- The PSP briefly holds KES between the two steps. Direct customer-to-merchant STK Push is not practical in Daraja: STK Push can only target shortcodes (excluding personal M-Pesa numbers) and requires per-shortcode API authorization that does not scale across merchants.

### BTC rail (Lightning)
- Merchant's phone runs an embedded Lightning node (candidate implementations: LDK-node, Breez SDK, or a PhoenixD variant; final choice pending) with keys generated and stored on device
- The operator's LSP provides inbound liquidity to the merchant's phone over an LSP protocol (candidate: LSPS2)
- Customer scans a Lightning invoice generated by the PoS; sats settle directly to the merchant's phone node
- The operator never holds sats at any point after the LSP hop
- On-chain Bitcoin receiving is out of scope for v1; seed derivation remains on-chain-compatible for a future release

Two rails, two ledgers, no automatic conversion in v1. Cross-rail conversion (BTC-to-M-Pesa cashout, M-Pesa-to-BTC top-up) is a candidate for a future product.

## Corridor abstraction

Everything corridor-specific is behind an interface from day one. The reference implementation is Kenya:
- Currency: KES
- Mobile money PSP: Safaricom via Daraja
- Rate source: Trex, delivered via BTCPay's exchange rate plugin architecture
- Phone validation: Kenyan MSISDN format

Forkers implement:
- A `Corridor` config specifying currency, phone validation rules, KYC requirements, and default rate source
- A PSP adapter (Vodacom M-Pesa API for Tanzania, MTN for Uganda/Ghana, Airtel Money for various markets)
- Optionally a corridor-specific rate plugin

The mobile app compiles against a single active corridor per build. Multi-corridor within one app is out of scope for v1.

## Open source and distribution

The project is MIT licensed (see [LICENSE](LICENSE)). Distribution follows the BTCPay Server pattern:

- One canonical Flutter codebase in this repository
- Tando publishes "Tando PoS" from that codebase, pointing at the reference deployment's backend infrastructure (Bitcoin.co.ke LSP, Splice PSP, Trex rate plugin)
- Any operator can fork, rebrand, point at their own backend (own LSP, own PSP, own rate plugin), and publish their build via app store, direct APK distribution, or sideload
- No architectural dependency on any app store gatekeeper

Pluggable components:
- LSP (any node implementing the chosen LSP protocol; LSPS2 is the current candidate)
- Rate provider (any BTCPay exchange rate plugin)
- PSP (any conforming corridor adapter)
- Cloud key backup provider

## Mobile app

### Technology
- Flutter, cross-platform. Android is the primary target given the East African smartphone mix; iOS is supported
- Embedded Lightning node on device (candidate implementations: LDK-node with FFI to the Rust core, Breez SDK Flutter bindings, or a PhoenixD variant; final choice pending)
- OS-native key storage: Android Keystore, iOS Keychain
- Foreground-required for receiving payments: Android via foreground service with persistent notification; iOS via PushKit VOIP wakes as an optimization
- Kiosk / immersive mode during payment flow to keep the app foregrounded

### App topology
The app has two surfaces gated by authentication level:

**Owner surface** (requires owner biometric/PIN):
- Store settings and M-Pesa identifier configuration
- Wallet: BTC balance, transaction history, spend/send actions
- Reports: sales history, per-employee attribution
- Staff management: invite/revoke employees
- Node health (shown as reassurance, not as an actionable surface)
- Backup and recovery

**PoS surface** (default post-login, no elevated auth):
- Amount entry or product selection from the BTCPay store inventory
- Payment method selection screen (BTC/KES with per-method price)
- Payment execution (STK Push initiation or Lightning invoice display)
- Payment confirmation
- No balance display, no settings, no wallet actions

### Multi-account, single-device
Multiple accounts (one owner plus zero or more employees) can be installed side-by-side on one device. Startup shows an account picker with the last-active profile pre-selected. Each account has its own PIN or biometric. Switching between profiles is sub-5-second.

The embedded Lightning node runs at the app/OS process level and remains online regardless of which account is currently active. Employees can create invoices and receive payments (autonomous node operations, no unlock needed). Signing operations that move funds outbound (spends, channel closes, seed export) require owner unlock.

### Roles and permissions
Users map 1:1 to BTCPay store users. Each merchant store has one Owner (holds the node keys, sees full app) and zero or more Employees (permissioned via BTCPay's native role model, only see the PoS surface).

Invite flow:
- Owner opens Owner surface, taps "Add employee," enters employee phone number, sends invite
- Employee gets SMS with a link to install the app
- Employee opens app, signs in with their phone, gets OTP, lands in PoS surface only
- Owner sees list of employees, can revoke any at any time, sees per-transaction attribution via BTCPay's native audit trail

### Connectivity awareness
The app polls the operator's services health endpoint at a low frequency (implementation likely CDN-fronted health JSON to keep origin load flat at scale). A status indicator shows green (healthy) or red (unreachable) in a consistent location. Immediately before every payment initiation, the app performs a direct real-time health check to fail fast if needed.

Rationale: Kenyan mobile data is not always reliable; the app helps merchants self-diagnose "am I online" without ambiguity.

## Merchant onboarding

Onboarding follows a "yes ladder" philosophy: minimize friction and financial commitment at signup so merchants complete the flow, then introduce paid services only after they've invested time and generated activity.

High-level steps (a detailed screen-by-screen walkthrough is a TBD open item):

1. Install app
2. Sign up with phone number and OTP verification
3. User provides their email address
4. Backend creates BTCPay store (default store name = phone number; default currency = corridor's; default rate source = operator's rate plugin)
5. Embedded Lightning node initialization: seed generation, forced key backup with cloud storage unlocked by passkey
6. Optional M-Pesa identifier entry (can be deferred to settings)
7. Ready to accept payments (first Lightning payment triggers a just-in-time channel open, absorbed by the operator; merchant sees a brief "setting up your first payment" moment)

## LSP model

The operator provides a Lightning Service Provider (LSP) node that provisions inbound liquidity to merchant phone nodes over a standardized LSP protocol (candidate: LSPS2). In the reference deployment the LSP is Bitcoin.co.ke, running LND; any LSP-capable node works. Forkers substitute their own LSP.

### Channel provisioning strategy (hybrid)

- **First payment (JIT):** No pre-provisioned channel. When the first customer scans a Lightning invoice, the LSP opens a channel just-in-time. Merchant sees a brief "Setting up your first payment" moment. The operator absorbs this first channel as customer acquisition cost, in line with the yes-ladder onboarding.
- **Post-activation (persistent):** Background job monitors merchant activity. Once a threshold is crossed (specific thresholds TBD, currently proposed at 5+ transactions or 50k+ sats received), the LSP opens a persistent channel sized to observed volume.
- **Ongoing (auto-provision):** LSP watches each merchant's inbound liquidity. When it drops below a low-water threshold (~20% remaining), it opens supplementary inbound before the merchant experiences a receive failure at the till. Merchant sees this as an audit line in the Owner surface only; no action required.

Merchants never see channel opens, closes, splices, or liquidity numbers as decisions they must make. The Owner surface shows node health as reassurance ("your capacity was topped up automatically") not as a knob to turn.

### LSP economics
- The operator absorbs the first channel open (JIT) as customer acquisition cost. No money commitment at signup.
- Ongoing supplementary channel opens are folded into subscription
- Any additional paid LSP services are introduced only after the merchant is engaged and transacting
- Forkers set their own LSP economics for their deployments

## Rate quotation

Rates come from BTCPay's exchange rate plugin architecture. The reference deployment uses a Tando-maintained Trex-backed plugin as the default. Forkers select any BTCPay rate plugin (or ship their own).

- UI always shows KES amounts
- BTC sats amount is queried at the moment "Pay via Lightning" is selected (not at cart-build time)
- Lightning invoices expire in 5 minutes; regenerate with fresh rate if expired
- No FX slippage exposure to merchant on the KES rail (STK Push transacts KES natively)

## Pricing markup (per-merchant, KES rail only)

Merchant sets a global markup in Owner surface settings, either as a percent or a fixed KES amount. Markup applies to the total KES of the payment when KES is selected. BTC payments are unmarked.

- Example (5% markup): 200 KES cart → 200 KES via Lightning or 210 KES via M-Pesa
- Example (10 KES fixed markup): 200 KES cart → 200 KES via Lightning or 210 KES via M-Pesa
- Markup is shown as a visible line item on the payment method screen so the customer sees exactly what they are paying and why

Rationale: encourages BTC adoption while covering the merchant's M-Pesa transaction costs. v1 is a store-global rate; per-item markup is a future version.

## Inventory

v1 offloads inventory management to BTCPay's web dashboard. Merchant owners log into their BTCPay web UI to add and edit products (name, KES price, optional image/category). The mobile app reads from BTCPay's store API and displays the current inventory in the PoS surface for cart-building.

A future version will add native in-app inventory CRUD for owners; not v1.

## Business model (deferred to a later phase)

- Flat monthly Bitcoin-priced subscription per merchant, covering PoS access, M-Pesa collection, LSP services, and support (specifics deferred)
- Any paid services are introduced only after merchant engagement is established (yes-ladder philosophy), not at signup
- Refunds handling (deferred, revisit before merchant launch)

## Recovery UX

Merchant loses their phone: install the app on a new phone, sign in with the same phone number, receive OTP, app reconstructs the seed from cloud backup unlocked by passkey.

Corner case: what happens if the merchant loses the passkey too (Google/Apple account intact but passkey device gone) is TBD.
